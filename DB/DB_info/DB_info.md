# Архитектура базы данных (ATS / Outstaff)

## 📌 Концепция
База данных спроектирована по **гибридной (современной) модели** специально под нужды системы, где данные парсятся с помощью Больших Языковых Моделей (LLM):
1. **Строгая реляционная нормализация** для системных сущностей (пользователи, доступы, статусы, шаблоны).
2. **Документо-ориентированный подход (NoSQL внутри SQL)** для хранения распарсенных резюме кандидатов и конфигураций стилей.

В качестве СУБД используется **PostgreSQL**, так как он обладает мощным встроенным типом `JSONB` и поддержкой GIN-индексов для быстрого поиска по документам.

---

## 🗄️ Описание таблиц

### 1. `system_users` (Внутренний персонал)
Таблица с сотрудниками, имеющими доступ к системе. Кандидатов в этой таблице **нет**.
* `login` / `hashed_password` — данные для авторизации.
* `role` — уровень доступа (director, hunter, admin).
* **Связи**: HR может быть привязан к множеству кандидатов (One-to-Many).

### 2. `candidates` (Ресурсы / Соискатели)
Основная таблица для хранения карточек кандидатов.
* `hr_id` — какой сотрудник добавил/ведет этого кандидата.
* `current_status` — бизнес-статус (`new`, `bench`, `on_project`, `outstaff`). Позволяет мгновенно фильтровать тех, кто свободен для распределения.
* `parsed_json_profile` (Тип: `JSONB`) — "сердце" кандидата. Хранит глубоко вложенную структуру, полученную от Нейромодуля (Опыт, Навыки, Образование, Питч).

### 3. `resume_templates` (Шаблоны экспорта)
Таблица для хранения корпоративных форматов резюме.
* `company_name` — название формата (например, "Сбербанк", "Яндекс").
* `style_config` (Тип: `JSONB`) — конфигурация стилей и шрифтов (Style JSON).
* `docx_file_path` — путь к базовому файлу `.docx` на сервере.
* `is_active` — флаг актуальности (позволяет скрывать устаревшие шаблоны без удаления).

## 🧠 Почему используется `JSONB` для резюме?

1. **Защита от изменений схемы (Schema Evolution)**: Если завтра Нейромодуль начнет извлекать новые данные (например, "Ожидания по зарплате"), нам **не нужно** делать миграции БД. JSONB просто примет новый ключ.
2. **Отсутствие JOIN-ада**: Классическая 3НФ потребовала бы разбить профиль на 10 таблиц (навыки, образование, проекты). Сборка такого профиля для отображения во фронтенде была бы медленной. `JSONB` отдает весь профиль одним запросом.
3. **Подготовка под RAG**: В будущем в таблицу можно добавить колонку `embedding (vector)` для семантического поиска по резюме с помощью расширения `pgvector`.
